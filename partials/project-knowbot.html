Key Components
  • Ingestion Pipeline – Scrapes the official UniLibre site and prepares documents for ingestion.
  • Pre-chunked FAQ Pack – Adds curated FAQs with controlled chunk boundaries and metadata.
  • Embedding Model – Uses Amazon Titan Embeddings v2 for document chunks and user queries.
  • Bedrock Knowledge Base – Stores content and performs retrieval for each user question.
  • Vector Database – Uses Amazon OpenSearch as the vector store for semantic retrieval.
  • Reranking Strategy – Applies a lightweight rerank step in the orchestration layer to prioritize the best chunks.
  • Guardrails – Applies safety and policy controls to improve reliability.
  • LangGraph Flow – Orchestrates the “embed → retrieve → rerank → answer” workflow.
  • API Layer – Lambda + API Gateway exposes a REST endpoint for the Web UI.
  • RAGAS Evaluation – Batch evaluation pipeline (separate from the production path) measuring Faithfulness, Answer Relevancy, Context Precision, and Context Recall against a curated golden-question set; LLM-as-judge metrics use Amazon Nova Pro (separate from the answer model) for unbiased scoring.
Query Flow (Embed → Retrieve → Rerank → Answer)

1) Embed: The user question is embedded using Amazon Titan Embeddings v2.

2) Retrieve: Query the AWS Bedrock Knowledge Base, retrieving from Amazon OpenSearch (vector index). Returns top chunks with metadata.

3) Rerank: Lightweight reranking prioritizes relevance (similarity, FAQ priority, freshness, de-duplication).

4) Answer: Claude 3.5 on Amazon Bedrock generates grounded responses, with safety controls via Bedrock Guardrails.

RAGAS Evaluation

A batch evaluation pipeline — running separately from the production query path — measures retrieval and generation quality against a curated set of golden question–answer pairs derived from the UniLibre knowledge base. Metrics like Faithfulness and Answer Relevancy use an LLM-as-judge (Amazon Nova Pro on Amazon Bedrock) and require live model API calls; Context Precision and Context Recall are computed via embedding similarity.

Metric What it measures Target
Faithfulness Claims in the answer are supported by retrieved context ≥ 0.85
Answer Relevancy Answer addresses the actual question asked ≥ 0.80
Context Precision Signal-to-noise ratio of retrieved chunks ≥ 0.75
Context Recall Ground-truth information covered by retrieved chunks ≥ 0.80

Each evaluation run produces a scored dataset that is logged and compared against the previous baseline, so regressions in retrieval or generation quality are caught before any model or knowledge-base update is promoted to production.

Improving RAGAS Scores

When a metric falls below its target, the remediation strategy depends on which part of the pipeline is responsible — retrieval, generation, or chunking.

Metric Root cause Remediation
Faithfulness Model generates claims beyond the retrieved context Tighten the system prompt to enforce context-only answers; reduce top-k to lower noise; strengthen Bedrock Guardrails grounding policy
Answer Relevancy Answer drifts from the actual question Add a query-rewriting step before embedding; introduce intent classification to detect vague questions and ask for clarification
Context Precision Retrieved chunks contain too much irrelevant content Raise the reranking score threshold; use smaller, more focused chunks; apply metadata filters (topic, section) to narrow retrieval before vector search
Context Recall Ground-truth information missing from retrieved chunks Increase top-k; review FAQ chunk splits to avoid cutting relevant content across boundaries; enrich domain-specific terms before embedding if Titan v2 struggles with UniLibre vocabulary

Cross-cutting: Run RAGAS per topic category (admissions, academic calendar, fees) to pinpoint which areas underperform rather than relying on the global average alone. Expand the golden Q&A set for more representative signal, and always compare a full eval run before and after any change to avoid improving one metric while regressing another.

Description

KnowBot is a self-serve RAG chatbot implemented on AWS Bedrock Knowledge Base. It uses Amazon OpenSearch as the vector database and applies Bedrock Guardrails for safer and more consistent responses.

The system blends website content with a curated pre-chunked FAQ dataset to control chunk boundaries, overlap, metadata, and retrieval behavior.

The bot is exposed via Lambda + API Gateway. The end-to-end flow is orchestrated with LangGraph.

Tech stack: AWS Bedrock Knowledge Base, Titan Embeddings v2, Bedrock Guardrails, Claude 3.5, OpenSearch (Vector DB), LangGraph, Lambda, API Gateway, S3, Web Scraper / Crawler, Pre-chunked FAQ ingestion, RAGAS (Faithfulness · Answer Relevancy · Context Precision · Context Recall)
